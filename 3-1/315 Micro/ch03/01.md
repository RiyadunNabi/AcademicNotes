---
How many bytes can be addressed?
Depends on how many bits the processor uses for addresses.
The 8086 uses 20-bit addresses → 2²⁰ = 1,048,576 = 1 megabyte (1 MB) of memory. explain?
---
Excellent question, Riyad. **This is exactly the problem that led Intel to invent the segment:offset addressing scheme in the 8086.**

Let's build the intuition.

---

## Step 1: A 16-bit register can only store 16 bits

Suppose you have a 16-bit register:

```text
AX = 1111111111111111
```

In hexadecimal:

```text
FFFFh
```

Its maximum value is:

```text
FFFFh = 65535 = 2^16 - 1
```

So a 16-bit register can only represent addresses from

```text
0000h
to
FFFFh
```

which is **64 KB**.

---

## Step 2: But the 8086 has 1 MB of memory

The 8086 can access

```text
1 MB = 1,048,576 bytes
```

To uniquely identify every byte, it needs addresses from

```text
00000h
to
FFFFFh
```

Notice something?

```text
00000h  ← 5 hexadecimal digits
FFFFFh  ← 5 hexadecimal digits
```

Each hexadecimal digit represents 4 bits.

So:

```text
5 hex digits × 4 bits = 20 bits
```

The processor therefore needs **20-bit physical addresses**.

---

## Step 3: The problem

The CPU needs addresses like

```text
8A3F2h
```

which is **20 bits**.

But its registers are only 16 bits:

```text
FFFFh
```

You simply cannot store

```text
8A3F2h
```

inside a 16-bit register because it's too large.

Think of trying to write a **5-digit number** into a box that only has room for **4 digits**.

Example:

```
Need to store:
12345

Register can hold:
____
```

The last digit has nowhere to go.

---

## Step 4: Intel's clever solution

Instead of storing one 20-bit address, Intel split it into **two 16-bit values**:

```
Segment
Offset
```

Each fits perfectly into a 16-bit register.

For example:

```
Segment = 1234h
Offset  = 5678h
```

Both are 16 bits.

---

## Step 5: How do these become a 20-bit address?

The CPU computes the physical address using this formula:

[
\boxed{\text{Physical Address} = (\text{Segment} \times 16) + \text{Offset}}
]

Multiplying by 16 is the same as shifting the segment left by 4 bits.

Example:

```
Segment = 1234h
Offset  = 5678h
```

Shift the segment left by one hexadecimal digit:

```
12340h
```

Then add the offset:

```
12340h
+5678h
-------
179B8h
```

The result is:

```
179B8h
```

This is a **20-bit physical address**.

---

## Step 6: Which registers hold these values?

The 8086 has dedicated **segment registers**:

```
CS  (Code Segment)
DS  (Data Segment)
SS  (Stack Segment)
ES  (Extra Segment)
```

And it has **offset registers**, such as:

```
IP
SP
BP
SI
DI
BX
```

For example:

```
CS = 1234h
IP = 5678h
```

The CPU calculates:

```
Physical Address
=
12340h + 5678h
=
179B8h
```

---

## Big Picture

```
Memory (1 MB)

00000h
│
│
│
│
FFFFFh   ← needs 20 bits
```

But registers are only:

```
16 bits

0000h
to
FFFFh
```

So instead of one 20-bit register:

```
❌ 20-bit Address Register
```

the 8086 uses:

```
✔ Segment Register (16 bits)
✔ Offset Register  (16 bits)
```

The CPU combines them internally to produce the 20-bit physical address.

---

### Key idea to remember

> **The 8086 can address 1 MB of memory (20-bit addresses), but since all its registers are only 16 bits wide, it cannot store a full 20-bit address directly. Instead, it stores the address as two 16-bit parts—a segment and an offset—and combines them internally using the formula:**
>
> **Physical Address = (Segment × 16) + Offset**
